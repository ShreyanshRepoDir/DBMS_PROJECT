Real Estate Booking Platform Abstract

Abstract

The Real Estate Booking Platform is a comprehensive database management system designed to streamline the process of property listing, searching, and booking. The system serves three main user roles: Customers, Agents, and Administrators. By leveraging a robust PostgreSQL backend, an Express.js API, and a React frontend, the platform ensures data integrity, real-time availability checking, and secure transactional operations. Key database features such as triggers, stored procedures, and complex aggregations are implemented to enforce business logic directly at the database layer, ensuring optimal performance and consistency.

Real Estate Booking Platform Chapter 1

Chapter 1: Introduction

1.1 Overview of the Project

The project is a web-based Real Estate Booking Platform that connects property agents with potential customers. It provides a centralized database for managing property details, user roles, bookings, payments, and reviews.

1.2 Problem Statement

Traditional real estate processes heavily rely on manual paperwork and disjointed communication channels. There is a lack of a unified system that securely handles property bookings, avoids overlapping reservations, and provides analytical insights for administrators.

1.3 Objectives of the Project

  • To develop a relational database system that efficiently handles property and user data.
  • To implement strict constraints and triggers preventing duplicate or overlapping bookings.
  • To provide role-based access control for Customers, Agents, and Admins.
  • To demonstrate advanced SQL operations such as aggregations, joins, and transactions.

1.4 Scope of the Project

The scope includes the development of a full-stack prototype covering user authentication, property CRUD operations, booking and payment flows, and an administrative dashboard for system-wide analytics.

1.5 Existing System

Current manual systems or basic listing websites lack integrated booking and payment processing. They often require offline follow-ups and lack real-time synchronization, leading to double-booking errors.

1.6 Proposed System

The proposed system integrates the database layer directly with the application logic using automated triggers and procedures, ensuring that once a property is booked, its availability is instantly updated globally without manual intervention.

1.7 Advantages of Proposed System

High data integrity, real-time updates, secure transaction processing through PostgreSQL functions, and detailed administrative oversight.

1.8 Organization of the Report

This report is organized into six chapters covering Introduction, System Requirements, System Design, Implementation, Testing, and Results & Discussion.

Chapter 2: System Requirements and Specifications

2.1 Hardware Requirements

  • Processor: Intel Core i5 or equivalent
  • RAM: 8 GB minimum
  • Storage: 50 GB free disk space

2.2 Software Requirements

  • Operating System: Windows, macOS, or Linux
  • Database: PostgreSQL 16+
  • Backend: Node.js, Express.js
  • Frontend: React (Vite), Bootstrap 5

2.3 Functional Requirements

The system must allow users to register and log in. Agents must be able to list and edit properties. Customers must be able to browse properties, book them, and make simulated payments. Admins must be able to view all bookings and manage platform-wide properties.

2.4 Non-Functional Requirements

The system must respond within 2 seconds for search queries. Passwords must be securely hashed using bcrypt. The database must maintain ACID properties for all booking transactions.

2.5 Feasibility Study

The project is technically feasible as it utilizes open-source, widely supported web technologies. It is economically feasible due to the lack of licensing costs for PostgreSQL and Node.js.

Chapter 3: System Design

3.1 Architecture Diagram

The system follows a standard 3-tier architecture:

  • Presentation Tier: React SPA running in the client browser.
  • Application Tier: Node.js/Express REST API serving as the backend logic.
  • Data Tier: PostgreSQL database handling persistent storage and business logic triggers.

3.2 Data Flow Diagram (DFD)

Level 0 Context Diagram: The User (Customer/Agent/Admin) interacts with the Booking System, which in turn queries and updates the Database.

Level 1 DFD: Processes include Authentication, Property Management, Booking Processing, and Analytics Generation. Data flows from User Inputs -> Validation -> Database Storage -> Output Dashboards.

3.3 ER Diagram

Entities identified: Users, Agents, Properties, Bookings, Payments, Reviews, Favorites.
Relationships: - A User can have many Bookings (1:N).
- A Booking has one Payment (1:1).
- An Agent manages many Properties (1:N).
- A Property can have many Bookings (1:N).

3.4 Schema Diagram

The relational schema maps the ER diagram into strict tables with Foreign Key constraints enforcing referential integrity. ON DELETE CASCADE is utilized to maintain database cleanliness.

3.5 Normalization

All tables are normalized up to the Third Normal Form (3NF). There are no repeating groups (1NF), all non-key attributes are fully functionally dependent on the primary key (2NF), and there are no transitive dependencies (3NF).

Chapter 4: Implementation

4.1 Front-End Design

The frontend is implemented using React and Vite. Bootstrap 5 is used for responsive layouts. State management is handled via React Context API, specifically for Authentication tracking.

4.2 Back-End Design

The backend uses Express.js. The `pg` library is utilized to connect to PostgreSQL. Parameterized queries are strictly used to prevent SQL injection attacks. Middleware handles JWT validation and role-checking.

4.3 SQL Queries Used

Complex queries including JOINS and Aggregations are used heavily in the Admin Dashboard. For example:

SELECT p.city, COUNT(b.id) as booking_count 
FROM properties p 
JOIN bookings b ON p.id = b.property_id 
GROUP BY p.city 
ORDER BY booking_count DESC LIMIT 1;

4.4 Stored Procedures / Triggers

Trigger 1 (trg_booking_status_change): Automatically updates a property's status to 'booked' when a booking is confirmed.

Trigger 2 (trg_prevent_duplicate_booking): Raises an exception if a booking is attempted on dates that conflict with an active booking.

Function (confirm_booking): Wraps the insertion of a payment record and the updating of a booking status inside a single atomic database transaction.

Chapter 6: Results and Discussion

6.1 Output Screens

The final application presents a clean, user-friendly interface. The Admin Dashboard successfully consolidates complex database tables into readable statistics and management tables.

6.2 Performance Analysis

By implementing Database Indexes on `properties(city)` and `bookings(status)`, search filtering and dashboard aggregations perform efficiently even as the dataset grows.

6.3 Advantages of the System

The utilization of database-level triggers and functions drastically reduces the amount of code needed in the Node.js backend, centralizing business logic where data lives, thus guaranteeing consistency regardless of the client.

6.4 Limitations of the System

The current prototype utilizes a mocked payment system rather than integrating with a real-world gateway like Stripe.

6.5 Future Enhancements

Future development could include location-based mapping APIs, real email notifications for booking confirmations, and advanced date-range overlap calculations for properties.
